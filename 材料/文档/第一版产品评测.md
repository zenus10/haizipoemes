# HaiZiPoemes 产品评审报告

## 一、总体印象

这是一个致敬诗人海子（1964-1989）的诗歌档案网站，基于 Astro 静态框架构建，部署于 Netlify。收录了约 257 首诗歌（短诗 238 首、长诗 4 首、太阳七部书 7 篇、文论 8 篇）及 28 条生平事件。技术选型合理，实现质量中上，但在产品定位、内容体验和功能完整度方面存在明显的提升空间。

**综合评分：6.5 / 10**

------

## 二、优点（做得好的地方）

### 1. 技术选型正确

- Astro 静态生成 + Markdown 内容管理是诗歌类站点的理想方案：加载快、SEO 友好、维护成本低
- UnoCSS 原子化方案控制了 CSS 体积
- 零运行时 JS 框架依赖，纯静态输出

### 2. 工程细节有亮点

- 虚拟滚动列表（40px 行高 + 5 条 overscan）处理 250+ 首诗无性能问题
- 筛选状态持久化到 SessionStorage，刷新不丢失
- 搜索防抖 150ms、滚动用 rAF、事件监听 passive——性能意识到位
- 目录组件（TOC Web Component）有三态追踪（past/active/future）+ 进度条，阅读体验不错

### 3. 内容收录量可观

- 短诗按年份（1983-1989）分类组织，覆盖较完整
- 长诗和太阳七部书这类万字级作品也做了收录
- 生平线 28 个节点，覆盖出生到身后

### 4. CSS 变量体系规范

- 颜色、间距、字体、阴影、圆角、动画全部变量化
- 为后续主题扩展（如暗色模式）打了基础

------

## 三、问题与不足（需要直面的短板）

### A. 产品定位模糊——"这个网站为谁而建？"

这是最核心的问题。网站目前处于"诗歌列表 + 年表"的工具状态，缺乏明确的产品叙事。

- **首页只有一张背景图**，没有任何文字引导、诗歌推荐或情感入口。用户打开网站后不知道该做什么。一个致敬型网站的首页应该在 3 秒内传达"这是谁、为什么重要、你该从哪里开始"
- 没有"关于海子"的独立介绍页——Timeline 页面是事件列表而非传记叙事，新读者无法快速建立认知
- 缺少编辑策略：哪些是代表作？哪些适合入门？哪些是被低估的佳作？目前 250+ 首诗以平等权重堆叠，对普通读者极不友好

### B. 阅读体验——核心场景打磨不够

诗歌网站的核心是"读诗"，但这里的阅读体验有几个硬伤：

1. **诗歌详情页缺乏上下文**：读完一首诗后没有"上一首/下一首"导航，没有同年份/同类型推荐，阅读流被强制中断
2. **长诗排版问题**：太阳·土地篇（3000+ 行）和太阳·弑这类超长作品，每行之间都有空行，导致页面极长。长诗需要不同于短诗的排版策略（分段锚点、章节跳转、折叠等）
3. **没有收藏/标记功能**：读者无法标记喜欢的诗，无法建立个人阅读记录
4. **字体方案存在问题**：正文用了 Satoshi（西文字体）作主字体，中文回落到微软雅黑/苹方——但这是一个纯中文内容站点，应该以中文字体为主导。建议用思源宋体（Noto Serif SC）或霞鹜文楷作为诗歌正文字体，更契合诗歌的文学气质

### C. 导航与信息架构

1. **移动端导航缺失**：640px 以下直接隐藏了导航链接（`display: none`），没有汉堡菜单或其他替代方案。移动端用户完全无法导航
2. **筛选器缺少标签维度**：有 type 和 year 筛选，但没有利用 tags 字段。海子诗歌有丰富的主题维度（麦地、村庄、死亡、太阳、远方等），按主题筛选对读者价值远大于按年份
3. **搜索只支持标题**：不支持全文搜索。用户想找"面朝大海"可以，但想找包含"麦子"的所有诗就做不到

### D. 视觉设计

1. **配色单调**：深红（#600000）作为唯一强调色，在大面积白底上显得沉闷。一个纪念性网站可以沉稳，但不该沉闷。建议引入暖灰、米色等辅助色温暖画面
2. **没有暗色模式**：2026 年的网站产品，暗色模式已是基本预期
3. **Timeline 页面的卡片堆叠交互**在桌面端有创意，但在信息展示效率上不如传统纵向时间轴——用户需要不断点击切换，无法一目了然地纵览全貌
4. **Contact 页面的存在意义不明**：一个诗歌档案站为什么需要联系表单？如果是用于勘误反馈，应该明确标注用途

### E. 内容质量

1. **长诗格式问题**：之前的对话中已经在修复太阳·弑和太阳·土地篇的格式（表格残留、行连接等），说明原始内容录入质量不够稳定。建议对所有内容做一次系统校对
2. **部分元数据不一致**：比如"河流"标注为 `type: "long"` 而非 `type: "sun"`，但它实际属于太阳七部书体系。分类准确性影响筛选可靠性
3. **没有注释/解读层**：海子的诗（尤其长诗）晦涩难懂，缺少任何辅助理解的注释、背景说明或学术解读引用

### F. SEO 与可发现性

1. **Meta description 全站统一为"海子的诗"**——每个页面应有独立的描述
2. **没有 Open Graph / Twitter Card 标签**：分享到社交媒体时无法显示预览图和摘要
3. **没有 sitemap.xml**：影响搜索引擎索引效率
4. **没有 structured data**（JSON-LD）：可以标注为 CreativeWork/Poem 类型，提升搜索结果展示

### G. 可靠性与维护

1. **零测试覆盖**：没有任何单元测试、集成测试或 E2E 测试
2. **Contact 表单的 Formspree ID 是占位符**，实际不可用
3. **Netlify 配置中 SPA fallback（`/\* → /index.html`）与 Astro 静态生成矛盾**——Astro 生成的是 MPA（多页面应用），不需要 SPA 路由回退。这个配置可能导致 404 页面无法正常工作

## 六、总结

这个项目在技术实现上展现了不错的工程能力（虚拟滚动、Web Components、状态持久化等），但产品思维和内容策略方面还比较薄弱。最大的问题不是"做得不好"，而是"不知道为谁而做"——它目前更像一个开发者的技术练习，而非一个能让读者产生情感连接的诗歌体验。

海子的诗本身有强大的感染力，网站应该做的是**让这种感染力自然流淌**，而不是把 250 首诗堆进一个列表让读者自己翻找。好的诗歌网站应该像一个策展人，引导读者走进诗人的世界。

核心建议：**先解决"首页空白 + 移动端无导航 + 无阅读流"这三个基本体验问题**，然后围绕"帮助读者发现好诗"这个目标逐步迭代。
